# 0.15.0

## New Components & APIs

- **`defineTool()`** — First-class custom tool authoring primitive exported from the root package. Wraps the Vercel AI SDK `tool()` with automatic Smithers event logging, idempotency key injection, and side-effect metadata tracking. Tools declared with `sideEffect: true` receive a `ctx.idempotencyKey` for safe retries.
  ```ts
  import { defineTool } from "smithers";

  const deploy = defineTool({
    name: "deploy",
    description: "Push build to production",
    schema: z.object({ env: z.string() }),
    sideEffect: true,
    async execute(args, ctx) {
      // ctx.idempotencyKey is stable across retries
      return await pushBuild(args.env, ctx.idempotencyKey);
    },
  });
  ```
- **Task-level `allowTools`** — The `<Task>` component now accepts an `allowTools` prop to restrict which CLI tools an agent can invoke. Combined with the new `cliAgentToolsDefault: "explicit-only"` runtime config, this gives fine-grained control over agent capabilities per node.
  ```tsx
  <Task id="lint-fix" allowTools={["read_file", "edit_file", "bash"]} output={outputs.fix}>
    Fix the linting errors
  </Task>
  ```

## CLI

- **`smithers signal <run_id> <signal_name>`** — Deliver structured data to a running workflow's `<WaitForEvent>` node directly from the command line. Validates and persists the payload, then provides follow-up CTAs for resuming the run.
  ```bash
  smithers signal abc123 payment-received --data '{"amount": 99.95, "currency": "USD"}'
  ```
- **`smithers retry-task <workflow>`** — Retry a specific task within a run without time travel. Resets the target node and its dependents to `pending`, clears their output rows, and resumes the workflow. Only the reset tasks re-execute; completed tasks use cached results.
  ```bash
  smithers retry-task my-workflow.tsx -r abc123 -n implement
  smithers retry-task my-workflow.tsx -r abc123 -n analyze --no-deps true
  ```
- **`smithers timetravel <workflow>`** — Revert a run to a previous task state. Resets the target node and its dependents in the database, optionally restores the jj/git filesystem to its snapshot revision, and can immediately resume execution with `--resume`.
  ```bash
  smithers timetravel my-workflow.tsx -r abc123 -n review-step --resume
  ```

## New Run Status

- `waiting-event` — Run is paused on a `<WaitForEvent>` node, waiting for an external signal delivered via `smithers signal` or the gateway `signals.send` RPC.

## Gateway

- **WebSocket transport** — The gateway server now supports persistent WebSocket connections alongside HTTP. Clients can subscribe to run progress events in real time and send RPC messages over a single socket with protocol-level heartbeats.
- **JWT authentication** — A new `jwt` auth mode verifies HS256/HS384/HS512 tokens with configurable issuer, audience, and expiry validation. Roles and scopes are extracted from the token payload.
- **Cron triggers** — Register recurring workflow executions with standard cron expressions. The gateway evaluates due schedules on each supervisor poll and starts runs automatically.
  ```ts
  gateway.registerWorkflow("nightly-report", {
    schedule: "0 2 * * *",
    // ...
  });
  ```
- **Typed approval modes** — Approval requests now carry a `mode` field (`decision`, `select`, `rank`, or `gate`) and structured `options` for richer human-in-the-loop interactions.
- **Auto-approval** — Workflow registrations can include an `autoApprove` policy object. When present, matching approval requests are resolved immediately without human intervention.
- **Scoped approvals** — Approval requests can declare `allowedScopes` and `allowedUsers` to restrict who is authorized to approve or deny.

## Durability

- **Infinite retry by default** — Tasks now retry indefinitely with aggressive exponential backoff (1s → 2s → 4s → 8s → ... capped at 5 minutes) instead of failing after one attempt. This makes workflows durable by default: transient failures, rate limits, and model errors are absorbed automatically.
  ```tsx
  {/* This task retries forever until it succeeds */}
  <Task id="analyze" output={outputs.analysis} agent={analyst}>
    Analyze the codebase.
  </Task>

  {/* Opt out with noRetry or retries={0} */}
  <Task id="quick-check" output={outputs.check} agent={checker} noRetry>
    One-shot validation.
  </Task>
  ```
- **`noRetry` prop** — Explicit opt-out for tasks that should fail immediately. Equivalent to `retries={0}`. Use for tasks where retrying makes no sense (e.g., validation checks, idempotency-sensitive operations).
- **5-minute backoff cap** — Exponential backoff maxes out at 300 seconds to prevent unbounded delay growth.

## Engine & Infrastructure

- **Effect task bridge architecture** — The entire task execution layer has been rewritten around a bridge abstraction (`compute-task-bridge`, `static-task-bridge`, `deferred-bridge`, `deferred-state-bridge`). Tasks now route through typed Effect pipelines with retry semantics, deferred resolution for timers and signals, and a clean separation between compute tasks, static tasks, and deferred waits.
- **Activity bridge (Phase 1)** — `executeTask` is wrapped in an Effect Activity for structured retries, heartbeat propagation, and future replay determinism.
- **Side-effect tool call warnings** — When a task retries after a previous attempt already invoked non-idempotent side-effect tools, the agent prompt now includes an explicit warning that those side effects may have already occurred.
- **Runtime CLI tool defaults** — The `cliAgentToolsDefault` option (`"all"` or `"explicit-only"`) is enforced at the engine level, propagated through the run config, and respected by every `<Task>`.
- **Snapshot restore** — The engine now exposes helpers for restoring run state from time-travel snapshots, enabling `smithers timetravel` and `smithers replay` to cleanly reset nodes and their dependents.
- **Zod v3 compat layer removed** — The `zodV3Compat.ts` shim has been deleted. All schema handling now uses Zod v4 natively. Projects still on Zod v3 should upgrade before adopting 0.15.0.

## Error Codes

- **Added**: `EXTERNAL_BUILD_FAILED`, `SCHEMA_DISCOVERY_FAILED`
