# 0.15.0

## Python SDK

- **`smithers-py`** — A new Python package (`packages/smithers-py/`) that lets you define workflows in Python. Nodes are decorated functions (`@task`, `@gate`, `@loop`) that communicate with the smithers engine over a subprocess JSON protocol. Pydantic models are automatically converted to Zod schemas for output validation.
  ```python
  from smithers import task, gate, SmithersWorkflow

  @task(output_schema=ReviewResult)
  def review_code(ctx):
      return ReviewResult(approved=True, comments=[])

  workflow = SmithersWorkflow("code-review", [review_code])
  ```
  Python workflows are executed via the new external subprocess bridge (`src/external/python-subprocess.ts`) and can be run with `smithers run workflow.py`.

## New Components & APIs

- **`defineTool()`** — First-class custom tool authoring primitive exported from the root package. Wraps the Vercel AI SDK `tool()` with automatic Smithers event logging, idempotency key injection, and side-effect metadata tracking. Tools declared with `sideEffect: true` receive a `ctx.idempotencyKey` for safe retries.
  ```ts
  import { defineTool } from "smithers";

  const deploy = defineTool({
    name: "deploy",
    description: "Push build to production",
    schema: z.object({ env: z.string() }),
    sideEffect: true,
    async execute(args, ctx) {
      // ctx.idempotencyKey is stable across retries
      return await pushBuild(args.env, ctx.idempotencyKey);
    },
  });
  ```
- **Task-level `allowTools`** — The `<Task>` component now accepts an `allowTools` prop to restrict which CLI tools an agent can invoke. Combined with the new `cliAgentToolsDefault: "explicit-only"` runtime config, this gives fine-grained control over agent capabilities per node.
  ```tsx
  <Task id="lint-fix" allowTools={["read_file", "edit_file", "bash"]} output={outputs.fix}>
    Fix the linting errors
  </Task>
  ```

## CLI

- **`smithers signal <run_id> <signal_name>`** — Deliver structured data to a running workflow's `<WaitForEvent>` node directly from the command line. Validates and persists the payload, then provides follow-up CTAs for resuming the run.
  ```bash
  smithers signal abc123 payment-received --data '{"amount": 99.95, "currency": "USD"}'
  ```
- **`smithers retry-task <workflow>`** — Retry a specific task within a run without time travel. Resets the target node and its dependents to `pending`, clears their output rows, and resumes the workflow. Only the reset tasks re-execute; completed tasks use cached results.
  ```bash
  smithers retry-task my-workflow.tsx -r abc123 -n implement
  smithers retry-task my-workflow.tsx -r abc123 -n analyze --no-deps true
  ```
- **`smithers timetravel <workflow>`** — Revert a run to a previous task state. Resets the target node and its dependents in the database, optionally restores the jj/git filesystem to its snapshot revision, and can immediately resume execution with `--resume`.
  ```bash
  smithers timetravel my-workflow.tsx -r abc123 -n review-step --resume
  ```

## New Run Status

- `waiting-event` — Run is paused on a `<WaitForEvent>` node, waiting for an external signal delivered via `smithers signal` or the gateway `signals.send` RPC.

## Gateway

The gateway is the headless server that turns smithers into a remotely controllable workflow engine — the foundation for building bots, CI integrations, and autonomous agents that respond to external events (similar to Claude Bot or OpenClaw). Instead of running workflows from the CLI, external systems connect over WebSocket or HTTP, start runs, subscribe to progress, approve or deny human-in-the-loop gates, and deliver signals — all programmatically.

- **WebSocket RPC transport** — Persistent connections with a typed request/response/event frame protocol. Clients send JSON-RPC-style method calls (`runs.create`, `runs.list`, `approvals.decide`, `signals.send`, etc.) and receive real-time event streams for subscribed runs. Protocol-level heartbeats detect stale connections.
  ```ts
  import { SmithersGateway } from "smithers-orchestrator/gateway";

  const gateway = new SmithersGateway({ auth: { mode: "jwt", ... } });
  gateway.registerWorkflow("pr-review", prReviewWorkflow);
  gateway.listen(3000);
  ```
- **HTTP fallback** — Every RPC method is also available via `POST /rpc` with a JSON body, so stateless integrations (webhooks, GitHub Actions, cloud functions) can trigger workflows without maintaining a socket.
- **Role-based access control** — Every RPC method has an access level (`read`, `execute`, `approve`, `admin`). Connections are authenticated and scoped so a webhook endpoint can start runs but not approve them, while a human dashboard can do both.
- **Three auth modes**:
  - **`token`** — Static bearer tokens mapped to roles and scopes. Simple for internal services.
  - **`jwt`** — HS256/HS384/HS512 verification with configurable issuer, audience, expiry, and claim extraction. Roles and scopes are pulled from the token payload.
  - **`trusted-proxy`** — For deployments behind a reverse proxy that handles auth. Extracts identity from trusted headers.
- **Cron triggers** — Register recurring workflow executions with standard cron expressions. The gateway evaluates due schedules on each supervisor poll and starts runs automatically.
  ```ts
  gateway.registerWorkflow("nightly-report", {
    schedule: "0 2 * * *",
    // ...
  });
  ```
- **Typed approval modes** — Approval requests carry a `mode` field (`decision`, `select`, `rank`, or `gate`) and structured `options` for richer human-in-the-loop interactions from external UIs.
- **Auto-approval policies** — Workflow registrations can include an `autoApprove` policy object. When present, matching approval requests are resolved immediately without human intervention — essential for fully autonomous bot workflows.
- **Scoped approvals** — Approval requests can declare `allowedScopes` and `allowedUsers` to restrict who is authorized to approve or deny, enforced at the gateway level.
- **Run lifecycle management** — Full RPC surface for `runs.create`, `runs.cancel`, `runs.rerun`, `runs.get`, `runs.list`, `runs.diff`, `frames.list`, `frames.get`, `attempts.list`, `attempts.get`, `signals.send`, `approvals.list`, `approvals.decide`, and cron management (`cron.list`, `cron.add`, `cron.remove`, `cron.trigger`).

## Durability

- **Infinite retry by default** — Tasks now retry indefinitely with aggressive exponential backoff (1s → 2s → 4s → 8s → ... capped at 5 minutes) instead of failing after one attempt. This makes workflows durable by default: transient failures, rate limits, and model errors are absorbed automatically.
  ```tsx
  {/* This task retries forever until it succeeds */}
  <Task id="analyze" output={outputs.analysis} agent={analyst}>
    Analyze the codebase.
  </Task>

  {/* Opt out with noRetry or retries={0} */}
  <Task id="quick-check" output={outputs.check} agent={checker} noRetry>
    One-shot validation.
  </Task>
  ```
- **`noRetry` prop** — Explicit opt-out for tasks that should fail immediately. Equivalent to `retries={0}`. Use for tasks where retrying makes no sense (e.g., validation checks, idempotency-sensitive operations).
- **5-minute backoff cap** — Exponential backoff maxes out at 300 seconds to prevent unbounded delay growth.

## Engine & Infrastructure

- **Effect task bridge architecture** — The entire task execution layer has been rewritten around a bridge abstraction (`compute-task-bridge`, `static-task-bridge`, `deferred-bridge`, `deferred-state-bridge`). Tasks now route through typed Effect pipelines with retry semantics, deferred resolution for timers and signals, and a clean separation between compute tasks, static tasks, and deferred waits.
- **Activity bridge (Phase 1)** — `executeTask` is wrapped in an Effect Activity for structured retries, heartbeat propagation, and future replay determinism.
- **DurableDeferred bridge (Phase 2)** — Approvals and `WaitForEvent` nodes are now backed by `DurableDeferred` from `@effect/workflow`. When an approval decision is made or a signal is delivered, the existing DB records are persisted as before, then the DurableDeferred is resolved — giving these waits replay-safe resolution semantics. The CLI and HTTP API are unchanged; only the underlying engine plumbing is replaced.
- **Workflow.make bridge (Phase 3)** — The React scheduler loop is now wrapped in `Workflow.make()` from `@effect/workflow`. Child workflows route through `Workflow.execute()` with parent linkage, and scheduler wakeups use a buffered notifier queue instead of explicit `Promise.race` waits. This lays the groundwork for full replay determinism and crash recovery.
- **`usePatched()` hook** — Workflow versioning primitive exported from the root package. Returns `true` for new runs and `false` for old runs replaying against a prior version, enabling safe rolling deployments of workflow code changes.
- **Side-effect tool call warnings** — When a task retries after a previous attempt already invoked non-idempotent side-effect tools, the agent prompt now includes an explicit warning that those side effects may have already occurred.
- **Runtime CLI tool defaults** — The `cliAgentToolsDefault` option (`"all"` or `"explicit-only"`) is enforced at the engine level, propagated through the run config, and respected by every `<Task>`.
- **Snapshot restore** — The engine now exposes helpers for restoring run state from time-travel snapshots, enabling `smithers timetravel` and `smithers replay` to cleanly reset nodes and their dependents.
- **Zod v3 compat layer removed** — The `zodV3Compat.ts` shim has been deleted. All schema handling now uses Zod v4 natively. Projects still on Zod v3 should upgrade before adopting 0.15.0.

## Run Status Reclassification

- **Stale runs surfaced as `continued`** — `getRun` and `listRuns` now check the heartbeat timestamp at read time. Runs whose heartbeat has gone stale are returned with status `"continued"` instead of `"running"`, so the CLI, dashboard, and API accurately reflect that the owning process is no longer active. Filtering by `status=running` also includes `continued` runs for backwards compatibility.

## Fixes

- **Input schema table generation** — `zodToTable` and `zodToCreateTableSQL` now correctly generate input tables with a `run_id`-only primary key when a Zod input schema is provided. Previously, input tables were created with the composite `(run_id, node_id, iteration)` key used by task output tables, causing inserts to fail.
- **OpenAI structured-output schema sanitization** — Extracted a reusable `sanitizeForOpenAI` utility (and `zodToOpenAISchema` helper) that fixes Zod v4 JSON Schema output for OpenAI's `response_format` constraints: missing `type: "object"` on passthrough schemas and bare `additionalProperties: {}` are now corrected automatically.
- **Skip `--output-schema` on Codex resume** — The `codex exec resume` command does not accept `--output-schema`. The flag is now omitted when resuming a session, preventing startup errors.
- **Increased default heartbeat timeouts** — Local and sandbox task heartbeat timeouts raised from 60s/120s to 300s/300s. CLI agents (Claude Code, Codex, etc.) can spend minutes reading files and thinking without producing stdout; the previous thresholds caused spurious aborts on complex tasks.
- **`picocolors` declared as dependency** — `picocolors` is imported in 5 source files but was missing from `package.json`, forcing users to manually install it. Now declared explicitly.
