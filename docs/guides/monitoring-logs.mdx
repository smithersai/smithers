Smithers exposes four observability surfaces:

- persisted lifecycle events
- structured logs
- OpenTelemetry spans
- Effect metrics exported over OTLP

The runtime already instruments workflow runs, nodes, tools, cache behavior, approvals, database access, hot reloads, HTTP requests, and Jujutsu commands. You can send those signals to any OTLP-compatible backend and you can consume the same observability context inside your own Effect services.

## Enable OpenTelemetry Export

Smithers ships a public observability layer:

```ts
import { Context, Effect, Layer, Schema } from "effect";
import { Model } from "@effect/sql";
import { Smithers } from "smithers-orchestrator";
import { createSmithersObservabilityLayer } from "smithers-orchestrator/observability";

const AppLive = Layer.mergeAll(
  Smithers.sqlite({ filename: "./smithers.db" }),
  AgentLive,
  createSmithersObservabilityLayer({
    enabled: true,
    endpoint: "http://localhost:4318",
    serviceName: "bugfix-worker",
    logFormat: "json",
  }),
);
```

If you prefer environment-based configuration, the runtime also reads:

```bash
export SMITHERS_OTEL_ENABLED=1
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318
export OTEL_SERVICE_NAME=bugfix-worker
export SMITHERS_LOG_FORMAT=json
export SMITHERS_LOG_LEVEL=info
```

Those variables are respected by the built-in runtime used by Smithers execution and the HTTP server.

## Local Prometheus + Grafana Stack

The repo ships a ready-made local stack under `observability/`:

```bash
docker compose -f observability/docker-compose.otel.yml up
```

That starts:

- OTLP collector on `http://localhost:4318`
- Prometheus on `http://localhost:9090`
- Grafana on `http://localhost:3000`
- Tempo on `http://localhost:3200`

The collector exports metrics on `:8889`, Prometheus scrapes that endpoint, and Grafana comes pre-provisioned with a Smithers dashboard.

If you use the built-in Smithers HTTP server, it also exposes a direct Prometheus endpoint at `GET /metrics`.

## Direct Prometheus Endpoint

The built-in server returns Prometheus text exposition on `/metrics`:

```bash
curl http://localhost:7331/metrics
```

If you want to expose the same metrics from a custom server, Smithers also exports a renderer:

```ts
import {
  prometheusContentType,
  renderPrometheusMetrics,
} from "smithers-orchestrator/observability";

const body = renderPrometheusMetrics();
// return body with Content-Type: prometheusContentType
```

## Built-in Smithers Metrics

Import the predeclared metrics from `smithers-orchestrator/observability`:

```ts
import { smithersMetrics } from "smithers-orchestrator/observability";

smithersMetrics.runsTotal;
smithersMetrics.nodeDuration;
smithersMetrics.httpRequestDuration;
```

The exported registry currently includes:

- `runsTotal`, `activeRuns`
- `nodesStarted`, `nodesFinished`, `nodesFailed`, `activeNodes`
- `nodeDuration`, `attemptDuration`
- `toolCallsTotal`, `toolDuration`
- `cacheHits`, `cacheMisses`
- `dbQueryDuration`, `dbRetries`
- `schedulerQueueDepth`
- `approvalsRequested`, `approvalsGranted`, `approvalsDenied`
- `httpRequests`, `httpRequestDuration`
- `hotReloads`, `hotReloadFailures`, `hotReloadDuration`
- `vcsDuration`

These metrics are emitted by the engine and runtime itself. You do not need to wire them by hand.

## Use Observability as a Dependency

The observability layer also provides a service tag you can depend on in your own Effect services:

```ts
import { Context, Effect, Layer, Metric } from "effect";
import { SmithersObservability } from "smithers-orchestrator/observability";

const notificationsSent = Metric.counter("app.notifications.sent");

class Notifications extends Context.Tag("Notifications")<
  Notifications,
  {
    readonly send: (ticketId: string) => Effect.Effect<void>;
  }
>() {}

const NotificationsLive = Layer.effect(
  Notifications,
  Effect.gen(function* () {
    const obs = yield* SmithersObservability;

    return {
      send: (ticketId) =>
        obs.withSpan(
          "notifications:send",
          Effect.gen(function* () {
            yield* obs.annotate({
              ticketId,
              channel: "slack",
            });
            yield* Metric.increment(notificationsSent);
            yield* Effect.logInfo(`sending notification for ${ticketId}`);
          }),
          { component: "notifications" },
        ),
    };
  }),
);
```

This is the intended composition model:

- use `createSmithersObservabilityLayer(...)` once in your app layer
- depend on `SmithersObservability` in your own services when you want Smithers-scoped spans or shared annotations
- use normal Effect `Metric` primitives for custom application metrics

## Events and Logs Still Matter

OpenTelemetry does not replace Smithers' durable event log. Every run still persists typed events and, when enabled, writes the NDJSON stream to:

```txt
.smithers/executions/<runId>/logs/stream.ndjson
```

That gives you two complementary views:

- OTLP spans and metrics for dashboards and tracing
- durable event history for replay, approvals, debugging, and auditability

You can still tail the event log directly:

```bash
tail -f .smithers/executions/<runId>/logs/stream.ndjson
```

And you can still inspect the SQLite event table:

```bash
sqlite3 smithers.db "SELECT seq, type, payload_json FROM _smithers_events WHERE run_id = '<id>' ORDER BY seq DESC LIMIT 20;"
```

For the event model itself, see [Events](/runtime/events).

## Server Instrumentation

If you expose workflows through the Smithers HTTP server, request handling is instrumented as well:

- every request increments `smithers.http.requests`
- request latency records into `smithers.http.request_duration_ms`
- request, workflow-load, and body-read spans flow through the same OTLP layer

That means the same local collector stack works for both embedded workflow execution and the hosted server process.

## What This Buys You

With OTLP enabled, existing Smithers spans become real traces and existing metric updates become dashboardable time series without changing workflow logic. The only code you usually add is:

- one app-level observability layer
- optional custom metrics for your own domain
- optional `SmithersObservability` usage inside services that need extra span structure
