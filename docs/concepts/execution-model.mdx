---
title: Execution Model
description: How Smithers turns a builder graph into a durable Effect workflow.
---

Smithers executes workflows as durable state machines built from a typed graph. The important split is:

- **workflow definition is pure**
- **step execution is effectful**
- **state transitions are durable**

That gives you deterministic orchestration without a render cycle.

## The Core Loop

Every workflow run follows the same high-level cycle:

```txt
1. Build      workflow definition -> typed graph
2. Materialize load execution state from SQLite
3. Select     find ready steps whose dependencies are satisfied
4. Execute    run each ready step as an Effect
5. Commit     persist outputs, attempts, events, and transitions
              -> repeat until terminal
```

Unlike the old JSX API, the workflow definition is not re-rendered after each step. The graph is already explicit.

## Phase 1: Build the Graph

The workflow graph can be produced from any of Smithers' three definition layers:

- **TOON** — a `.toon` file is compiled into a workflow graph at build time
- **Effect builder** — `Smithers.workflow(...).build(...)` constructs the graph in TypeScript
- **JSX** — the React renderer extracts `TaskDescriptor` objects from a component tree

All three produce the same internal graph structure. The rest of the execution model is identical regardless of which layer defined the workflow.

The builder API runs once to produce a workflow graph:

```ts
const Workflow = Smithers.workflow({
  name: "review-loop",
  input: Input,
}).build(($) => {
  const analyze = $.step("analyze", { output: Analysis, run: ... });
  const review = $.step("review", {
    output: Review,
    needs: { analysis: analyze },
    run: ...,
  });

  return $.sequence(analyze, review);
});
```

This phase is pure:

- it creates step handles
- it records dependency edges
- it records control-flow structure
- it records policies like retry, timeout, approval, and concurrency

No Effects run during graph construction.

## Phase 2: Materialize Execution State

When you call `Workflow.execute(input)`, Smithers:

1. validates the input against the declared `Schema`
2. allocates or resumes a durable execution id
3. loads persisted workflow state from SQLite
4. reconstructs the current step state machine

At this point Smithers knows:

- which steps have completed
- which outputs already exist
- which attempts failed
- which approvals are pending
- which loop or branch nodes are suspended

The graph is static. The execution state is what changes over time.

## Phase 3: Select Ready Steps

Smithers walks the graph and marks each step according to durable state.

Typical states are:

| State | Meaning |
| --- | --- |
| `pending` | Known to the workflow but not yet eligible to run |
| `ready` | All dependencies and guards are satisfied |
| `running` | The step Effect is currently executing |
| `waiting-approval` | Suspended on a durable approval gate |
| `completed` | Output has been persisted successfully |
| `failed` | Terminal failure after retry policy is exhausted |
| `cancelled` | Interrupted by the user or runtime |
| `skipped` | Not taken because of a branch or guard |

A step becomes `ready` only when:

- every entry in `needs` has completed
- any approval or guard condition is satisfied
- any loop or branch controller allows the step to proceed
- concurrency limits permit it to run

Because dependencies are explicit, Smithers does not need to infer readiness from a re-rendered tree.

## Phase 4: Execute Ready Steps

Each ready step runs as a normal `Effect`.

```ts
const fix = $.step("fix", {
  output: Fix,
  needs: { analysis },
  run: ({ analysis }) =>
    Effect.gen(function* () {
      const coder = yield* Coder;
      return yield* coder.fix(analysis);
    }),
});
```

The `run` function receives a typed execution context made from:

- validated workflow input
- resolved `needs` outputs
- Smithers execution metadata such as step id, attempt number, and execution id
- cancellation/timeout signals

The step body can then use the normal Effect toolkit:

- `Effect.gen`
- `Context.Tag`
- `Layer`
- `Schedule`
- logging and tracing
- resource scoping

Smithers should not invent a second effect system inside steps.

## Phase 5: Commit Durable Transitions

When a step finishes, Smithers commits the transition durably:

- persist the output model row
- persist attempt metadata
- persist lifecycle events
- update execution state
- unblock downstream dependents

If a step fails, Smithers persists the failure and consults the retry policy. If the retry policy allows another attempt, the step transitions back to `pending` or `ready`; otherwise it becomes `failed`.

This is where Smithers should lean on durable workflow primitives rather than hand-rolled bookkeeping.

## Determinism

Determinism in the new API comes from explicit structure.

### Stable step identity

Each step has an explicit logical id:

```ts
const analyze = $.step("analyze", { ... });
```

That id becomes the durable identity for persistence, logs, retries, approvals, and resume behavior.

### Explicit edges

Dependencies are declared directly:

```ts
const report = $.step("report", {
  needs: { research },
  run: ({ research }) => ...,
});
```

Smithers does not need to discover hidden dependencies by watching when user code reads from a shared context.

### Explicit control flow

Sequence, parallelism, branching, and looping should be represented as explicit builder nodes. That matters because runtime decisions must survive process restarts.

If a workflow has a review loop, the current iteration belongs in durable state, not in transient JavaScript control flow.

## Retry, Timeout, and Approval Semantics

These policies should map directly onto Effect primitives.

### Retries

Retries belong to `Schedule`, not a Smithers-only integer.

```ts
retry: Schedule.exponential("250 millis").pipe(
  Schedule.intersect(Schedule.recurs(3)),
)
```

That makes retry semantics consistent with the rest of an Effect application.

### Timeouts

Timeouts should accept `DurationInput` and be enforced around the step Effect. A timed-out step is persisted as a failed attempt and fed through the same retry policy.

### Approvals

Approvals should suspend the step durably. The important property is not UI integration, but recoverability: after a restart, the workflow must still know that a decision is pending.

## Persistence Model

Smithers should persist two categories of data:

### Domain outputs

These are your declared `Model`s:

```ts
class Review extends Model.Class<Review>("Review")({
  approved: Schema.Boolean,
  feedback: Schema.String,
}) {}
```

### Workflow metadata

This is owned by Smithers:

- executions
- step attempts
- retry history
- approvals
- loop state
- branch decisions
- event log

The user should not have to add workflow bookkeeping columns to every domain model.

## Resume and Replay

Resuming a workflow should mean:

1. load the execution by id
2. rebuild the graph definition
3. reload persisted execution state
4. continue from unfinished durable nodes

Completed steps should not re-run unless the user explicitly asks for replay or invalidates cached state.

That is the core contract of Smithers: workflow definitions are pure and replayable, but execution history is durable and incremental.

## Dynamic Workflows

Builder-based workflows still support dynamic behavior, but it has to be explicit.

Good examples:

- a branch node that chooses a subgraph from a previous step output
- a loop node that repeats until a persisted condition is met
- a queue or merge gate that limits concurrency durably

The unifying rule is that runtime control-flow decisions must be represented in durable state. They cannot depend on an ephemeral re-render of arbitrary user code.

## Why This Matters

This execution model makes the system easier to reason about:

- graph structure is explicit
- dependencies are typed
- runtime services stay inside Effect
- persistence is durable
- retries and approvals have a stable state machine

That is a better fit for workflow orchestration than reusing a UI renderer as the planner.
