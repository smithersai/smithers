title: Execution
description: How Smithers executes workflow graphs
---

The `run_graph` function executes a workflow graph, handling parallelism, caching, timeouts, and error recovery. All execution state is persisted to SQLite for constant visibility.

## Basic Execution

```python
from smithers import build_graph, run_graph, SqliteCache

graph = build_graph(final_workflow)
result = await run_graph(graph, cache=SqliteCache("./smithers.db"))
```

## Execution Model

Smithers executes workflows **level by level**, persisting state to SQLite at each step:

1. Run all Level 0 workflows in parallel
2. Wait for all to complete
3. Run all Level 1 workflows in parallel
4. Repeat until done

```python
# Graph levels:
# Level 0: [analyze]
# Level 1: [lint, test]  ← These run in parallel
# Level 2: [deploy]

# Execution timeline:
# t=0:  analyze starts
# t=10: analyze completes
# t=10: lint AND test start (parallel)
# t=20: both complete
# t=20: deploy starts
# t=25: deploy completes
```

## Options

```python
result = await run_graph(
    graph,
    cache=SqliteCache("./cache.db"),  # Enable caching
    max_concurrency=4,                 # Limit parallel workers
    timeout=300,                       # Global timeout in seconds
    node_timeout=60,                   # Default per-node timeout
)
```

### `cache`

Provide a cache to skip unchanged workflows:

```python
from smithers import SqliteCache

cache = SqliteCache("./smithers_cache.db")
result = await run_graph(graph, cache=cache)
```

### `max_concurrency`

Limit how many workflows run in parallel:

```python
# Even if Level 1 has 10 workflows, only 4 run at a time
result = await run_graph(graph, max_concurrency=4)
```

Useful for:
- Respecting API rate limits
- Managing memory usage
- Debugging parallel issues

### `timeout`

Set a global timeout for the entire graph execution:

```python
# Fail if entire run takes more than 5 minutes
result = await run_graph(graph, timeout=300)
```

### `node_timeout`

Set a default timeout for individual nodes:

```python
# Each node fails if it takes more than 60 seconds
result = await run_graph(graph, node_timeout=60)
```

## Data Flow

Outputs from upstream workflows are automatically passed to downstream:

```python
@workflow
async def analyze() -> AnalysisOutput:
    return await claude("Analyze", output=AnalysisOutput)

@workflow
async def implement(analysis: AnalysisOutput) -> ImplementOutput:
    # `analysis` is provided by run_graph from the analyze output
    return await claude(f"Fix {analysis.files}", output=ImplementOutput)
```

The executor:
1. Runs `analyze`
2. Stores the `AnalysisOutput` result
3. Runs `implement`, injecting the stored output as `analysis`

## Error Handling

When a workflow fails:

1. **The workflow raises an exception**
2. **Parallel siblings continue** (by default)
3. **Downstream workflows are skipped**
4. **The error is surfaced to the caller**

```python
try:
    result = await run_graph(graph)
except WorkflowError as e:
    print(f"Workflow {e.workflow_name} failed: {e}")
    print(f"Completed workflows: {e.completed}")
```

### Fail-Fast Mode

Stop all execution on first failure:

```python
result = await run_graph(graph, fail_fast=True)
```

## Progress Tracking (SQLite Events)

All execution events are persisted to SQLite for constant visibility:

```python
# Events are automatically logged to SQLite:
# - RunStarted, RunFinished, RunFailed
# - NodeReady, NodeStarted, NodeFinished, NodeCached, NodeFailed
# - LoopIterationStarted, LoopIterationFinished, LoopMaxIterationsReached
# - LLMCallStarted, LLMCallFinished
# - ToolCallStarted, ToolCallFinished

# Monitor with CLI:
# smithers watch ./smithers.db --run <run_id>

# Or query directly:
# SELECT * FROM events WHERE run_id = ? ORDER BY event_id
```

Optionally add in-process callbacks:

```python
async def on_progress(event: WorkflowEvent):
    if event.type == "started":
        print(f"Starting {event.workflow_name}")
    elif event.type == "completed":
        print(f"Completed {event.workflow_name} in {event.duration_ms}ms")

result = await run_graph(graph, on_progress=on_progress)
```

## Dry Run

Preview execution without actually running:

```python
plan = await run_graph(graph, dry_run=True)
print(f"Would execute: {plan.workflows}")
print(f"Estimated cost: ${plan.estimated_cost}")
```

## Execution Results

`run_graph` returns the output of the root workflow, but you can access more:

```python
result = await run_graph(graph, return_all=True)

# All workflow outputs
print(result.outputs)
# {
#     'analyze': AnalysisOutput(...),
#     'implement': ImplementOutput(...),
#     'deploy': DeployOutput(...),
# }

# Execution stats
print(result.stats)
# ExecutionStats(
#     total_duration_ms=15000,
#     workflows_executed=3,
#     workflows_cached=0,
#     tokens_used=5000,
# )
```

## Timeout Handling

Configure timeouts to prevent runaway execution:

### Per-Workflow Timeout

```python
from smithers import workflow, timeout
from datetime import timedelta

@workflow
@timeout(seconds=30)
async def quick_task() -> Output:
    ...

# Or with timedelta
@workflow
@timeout(timedelta(minutes=5))
async def longer_task() -> Output:
    ...
```

### Timeout Behavior

Configure what happens when a timeout occurs:

```python
# Default: fail the node
@workflow
@timeout(60, on_timeout="fail")
async def critical_task() -> Output:
    ...

# Skip the node instead of failing
@workflow
@timeout(60, on_timeout="skip")
async def optional_task() -> Output:
    ...
```

Timeout events (`NodeTimedOut`, `RunTimedOut`) are logged to SQLite for debugging.

## Real-Time Monitoring

### WebSocket Updates

Subscribe to real-time progress updates via WebSocket:

```python
from smithers import get_websocket_server

# Start the WebSocket server
server = get_websocket_server()
await server.start(host="localhost", port=8765)

# Run your graph - events are automatically broadcast
result = await run_graph(graph)

# Stop when done
await server.stop()
```

Client protocol (JSON over WebSocket):
- Subscribe to run: `{"action": "subscribe", "run_id": "run-123"}`
- Filter events: `{"action": "filter", "event_types": ["NodeStarted", "NodeFinished"]}`
- Ping/heartbeat: `{"action": "ping"}`

### Prometheus Metrics

Export metrics for production monitoring:

```python
from smithers import get_metrics_collector

collector = get_metrics_collector()
collector.attach_to_event_bus()

# Start HTTP server for Prometheus scraping
collector.start_server(port=9090)  # http://localhost:9090/metrics
```

Available metrics:
- `smithers_workflow_runs_total` - Total workflow runs by status
- `smithers_workflow_duration_seconds` - Workflow duration histogram
- `smithers_llm_calls_total` - LLM calls by model
- `smithers_llm_tokens_total` - Token usage by model

## Best Practices

<AccordionGroup>
  <Accordion title="Always use caching in production">
    Caching dramatically speeds up re-runs and saves API costs.
  </Accordion>

  <Accordion title="Set appropriate concurrency">
    Don't overwhelm APIs. Use `max_concurrency` to stay under rate limits.
  </Accordion>

  <Accordion title="Handle errors gracefully">
    Wrap `run_graph` in try/except and log failed workflows.
  </Accordion>

  <Accordion title="Set timeouts for all workflows">
    LLM calls can hang indefinitely. Always set reasonable timeouts.
  </Accordion>

  <Accordion title="Monitor in production">
    Use Prometheus metrics and WebSocket updates for observability.
  </Accordion>
</AccordionGroup>
