---
title: Data Model
description: How Smithers uses Effect Schema, Effect SQL Model, and internal workflow metadata.
---

The Smithers data model has three layers:

1. **workflow input** described with `Schema`
2. **domain outputs** described with `Model`
3. **execution metadata** owned by Smithers

Those layers serve different purposes and should stay separate.

## Input Uses `Schema`

Workflow input is the external boundary of the run.

**In TOON:**

```toon
input:
  ticketId: string
  description: string
```

**In the Effect builder:**

```ts
import { Schema } from "effect";

class Input extends Schema.Class<Input>("Input")({
  ticketId: Schema.String,
  description: Schema.String,
}) {}
```

Both compile to the same `Schema` validation. Smithers validates input against this schema before the workflow starts. After validation, steps receive a typed `input` value.

Use `Schema` for input because:

- it defines the runtime contract
- it supports validation and decoding
- it does not imply persistence strategy by itself

## Outputs Use `Model`

Step outputs are domain records that Smithers persists.

**In TOON:**

```toon
steps[1]:
  - id: analyze
    output:
      summary: string
      severity: "low" | "medium" | "high"
```

TOON output schemas compile to `Model.Class` definitions automatically.

**In the Effect builder:**

```ts
import { Model } from "@effect/sql";
import { Schema } from "effect";

class Analysis extends Model.Class<Analysis>("Analysis")({
  summary: Schema.String,
  severity: Schema.Literal("low", "medium", "high"),
}) {}

class Fix extends Model.Class<Fix>("Fix")({
  patch: Schema.String,
  explanation: Schema.String,
}) {}
```

The important point is that these are **domain models**, not workflow bookkeeping types.

You should not need to add fields like:

- `runId`
- `nodeId`
- `iteration`
- `attempt`
- `approvedAt`

Those belong to Smithers, not to your business data.

## Smithers Owns Workflow Metadata

To run durably, Smithers needs internal state that is distinct from user models:

- execution ids
- step ids
- attempt history
- retry state
- timeout state
- approval state
- loop iteration state
- branch decisions
- event journal

This data should live in framework-managed tables, not inside every output model.

That separation is the main difference from the old API, where the user had to think in terms of `(runId, nodeId, iteration)` rows.

## What Gets Persisted

When a step completes, Smithers should persist two things:

### The domain output

The user-declared `Model` value:

```ts
new Analysis({
  summary: "Token refresh silently fails after the first expiration window.",
  severity: "high",
})
```

### The execution transition

Framework state such as:

- which step produced the row
- which execution it belongs to
- which attempt produced it
- when it was written
- whether it unblocked downstream work

The domain model is for your app. The transition record is for orchestration.

## Step Identity and Output Identity

A workflow can use the same `Model` in multiple places:

```ts
const initialReview = $.step("initial-review", {
  output: Review,
  run: ...,
});

const finalReview = $.step("final-review", {
  output: Review,
  run: ...,
});
```

That means persistence cannot be keyed only by model name. Smithers must track at least:

- execution identity
- step identity
- model identity
- iteration or branch identity when relevant

This is another reason workflow metadata needs its own storage layer.

## Why `Model` Instead of Raw Tables

Using `Model` gives Smithers a better abstraction boundary than exposing raw SQL tables:

- the output type is explicit
- validation and encoding are shared
- repositories and queries can use the same Effect SQL model
- the user stays in domain types instead of persistence plumbing

The workflow definition should talk about `Analysis` and `Fix`, not about SQLite column names.

## Direct SQL Access Still Works

Smithers should not hide the SQL layer. If your app needs repositories or reporting queries, use Effect SQL directly against the same database.

```ts
import * as SqliteClient from "@effect/sql-sqlite-bun/SqliteClient";
import { Layer } from "effect";

const DatabaseLive = SqliteClient.layer({
  filename: "./smithers.db",
});
```

The intended architecture is:

- Smithers orchestrates workflow state
- Effect SQL handles model persistence and queries
- both run on the same SQLite client layer

## Input and Dependency Access Inside Steps

A step does not query tables directly in the normal path. Instead, Smithers resolves declared dependencies and passes them in:

```ts
const report = $.step("report", {
  output: Report,
  needs: { analysis },
  run: ({ input, analysis }) =>
    Effect.succeed(
      new Report({
        title: input.ticketId,
        body: analysis.summary,
        wordCount: 42,
      }),
    ),
});
```

That gives you:

- typed workflow input
- typed dependency outputs
- no raw SQL lookups in orchestration code

If you need ad hoc querying, do it explicitly through a repository or SQL service.

## Loop and Branch State

Loops and branches should persist their control-flow state separately from the domain outputs they produce.

Examples:

- current loop iteration
- whether a branch was taken
- which approval decision unblocked a step
- which queue slot is currently owned

These are orchestration facts, not business facts. They should not leak into your `Model` definitions.

## Migrations

Changing a `Model` is a schema change and should be treated as one.

Typical cases:

- adding a field to `Report`
- changing a field type in `Review`
- splitting one model into two models

Smithers should validate that persisted model schemas still match the workflow definition and either:

- apply a migration, or
- fail fast and require one

The workflow runtime should not silently reinterpret old rows with new model definitions.

## Suggested Mental Model

Use this rule of thumb:

- **`Schema`** answers: what input does this workflow accept?
- **`Model`** answers: what domain value does this step persist?
- **Smithers metadata** answers: what happened while the workflow executed?

If a field only matters to orchestration, it probably does not belong in your domain model.
